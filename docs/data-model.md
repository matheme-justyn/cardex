# Data Model & File Layout

This document describes Cardex's data storage structure.

---

## Storage Philosophy

Cardex uses a **hybrid storage model**:

- **SQLite**: Metadata and relationships (searchable, queryable)
- **Markdown files**: Notes and summaries (human-readable, Git-friendly)
- **PDFs**: Original papers (stored in user-managed folders)

---

## SQLite Schema

### papers

Main paper table storing metadata.

```sql
CREATE TABLE papers (
    id TEXT PRIMARY KEY,              -- SHA256 of original file
    title TEXT NOT NULL,
    authors TEXT,                     -- JSON array: ["Smith, J.", "Chen, L."]
    year INTEGER,
    venue TEXT,                       -- Journal or conference name
    venue_rank TEXT,                  -- e.g. "CORE A*", "Q1", "N/A"
    doi TEXT,
    file_path TEXT NOT NULL,          -- Relative to library root
    status TEXT DEFAULT 'unread',     -- unread / reading / done
    ocr_required INTEGER DEFAULT 0,   -- 0 or 1
    ingested_at TEXT NOT NULL,        -- ISO 8601 timestamp
    updated_at TEXT
);

CREATE INDEX idx_papers_year ON papers(year);
CREATE INDEX idx_papers_venue ON papers(venue);
CREATE INDEX idx_papers_status ON papers(status);
```

### summaries

Knowledge cards generated by Skills.

```sql
CREATE TABLE summaries (
    id TEXT PRIMARY KEY,              -- UUID
    paper_id TEXT NOT NULL,           -- FK → papers.id
    skill_name TEXT NOT NULL,         -- e.g. "methodology", "security_eval"
    content TEXT NOT NULL,            -- Markdown content
    generated_at TEXT NOT NULL,       -- ISO 8601 timestamp
    model TEXT,                       -- LLM model used (e.g. "gpt-4")
    FOREIGN KEY (paper_id) REFERENCES papers(id) ON DELETE CASCADE
);

CREATE INDEX idx_summaries_paper ON summaries(paper_id);
CREATE INDEX idx_summaries_skill ON summaries(skill_name);
```

### notes

User's freeform notes on papers.

```sql
CREATE TABLE notes (
    id TEXT PRIMARY KEY,              -- UUID
    paper_id TEXT NOT NULL,           -- FK → papers.id
    content TEXT NOT NULL,            -- Markdown content
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    FOREIGN KEY (paper_id) REFERENCES papers(id) ON DELETE CASCADE
);

CREATE INDEX idx_notes_paper ON notes(paper_id);
```

### citations

Citation relationships between papers.

```sql
CREATE TABLE citations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    citing_id TEXT NOT NULL,          -- FK → papers.id (paper that cites)
    cited_doi TEXT,                   -- DOI of cited paper
    cited_title TEXT,                 -- Title if DOI not available
    in_library INTEGER DEFAULT 0,     -- 0 = not ingested, 1 = in library
    citation_count INTEGER DEFAULT 1, -- How many papers cite this
    FOREIGN KEY (citing_id) REFERENCES papers(id) ON DELETE CASCADE
);

CREATE INDEX idx_citations_citing ON citations(citing_id);
CREATE INDEX idx_citations_cited_doi ON citations(cited_doi);
CREATE INDEX idx_citations_in_library ON citations(in_library);
```

## File System Layout

### Default Structure

```
~/Documents/cardex-library/           # User-configured root
├── papers/                           # Organized PDFs (after ingest)
│   ├── 2024/
│   │   ├── Nature/
│   │   │   └── Smith_2024_Quantum_Computing.pdf
│   │   ├── ICML/
│   │   │   └── Chen_2024_Neural_Architecture.pdf
│   │   └── arXiv/
│   │       └── Lee_2024_Preprint.pdf
│   └── 2023/
│       └── ...
├── inbox/                            # Watch folder (new PDFs drop here)
│   └── new_paper.pdf
├── needs_ocr/                        # PDFs flagged as unreadable
│   └── scan_without_text.pdf
└── markdown/                         # All Markdown content
    ├── papers/
    │   ├── smith_2024_quantum.md     # Main note for each paper
    │   └── chen_2024_neural.md
    ├── summaries/
    │   ├── smith_2024_quantum/
    │   │   ├── general.md            # Skill: general
    │   │   ├── methodology.md        # Skill: methodology
    │   │   └── security_eval.md      # Skill: security_eval
    │   └── chen_2024_neural/
    │       └── ...
    └── arguments/                    # Argue Engine outputs
        └── 2024-03-03-transformer-analysis.md
```

### Configuration

Users can customize the layout via `~/.cardex/config.yaml`:

```yaml
library:
  root_path: ~/Documents/research-papers
  pdf_folder: papers       # Where organized PDFs go
  inbox_folder: inbox      # Drop zone for new files
  ocr_folder: needs_ocr    # Unreadable PDFs
  markdown_folder: markdown  # All Markdown content
```

## Naming Strategy

File naming is controlled by a YAML strategy file (Phase 1).

**Example**: `config/naming_strategy.yaml`

```yaml
strategy: author_year_title
pattern: "{authors[0].last}_{year}_{title_clean}.pdf"
folder: "{year}/{venue_type}/{venue_name}/"
max_filename_length: 100
clean_title:
  lowercase: true
  remove_special_chars: true
  replace_spaces_with: "_"
```

**Result**:
```
Input:  "Neural Networks for Quantum Computing (ICML 2024)"
Output: papers/2024/ICML/Smith_2024_neural_networks_for_quantum.pdf
```

## Markdown Format

### Paper Note (`markdown/papers/{paper}.md`)

```markdown
---
title: "Smith et al., 2024 - Quantum Computing"
venue: Nature
venue_rank: "Tier 1 (CORE A*)"
year: 2024
doi: "10.1038/nature12345"
tags: [quantum, security, reading]
cardex_id: "abc123"  # SHA256 of PDF
---

# Smith et al., 2024 - Quantum Computing

**Status**: #reading | **Evidence Tier**: 1

## Key Findings
- New quantum-safe algorithm
- Outperforms [[chen_2023_neural|Chen 2023]] by 30%

## My Notes
- [ ] Compare with [[lee_2022_security]]
- Potential application in [[projects/thesis-chapter-3]]

## Cardex Summaries
- [[summaries/smith_2024_quantum/general|General Summary]]
- [[summaries/smith_2024_quantum/methodology|Methodology Analysis]]

## Links
- **Cites**: [[lee_2022_security]], [[wang_2023_crypto]]
- **Cited by**: [[jones_2024_quantum]] (in library)
```

### Skill Summary (`markdown/summaries/{paper}/{skill}.md`)

```markdown
---
paper_id: "abc123"
skill: methodology
model: gpt-4
generated_at: "2024-03-03T10:30:00Z"
---

# Methodology Analysis: Smith et al., 2024

## Research Design
- **Type**: Experimental study with control group
- **Sample**: 1000 quantum systems, 10 configurations each

## Datasets
- Custom quantum benchmark suite
- Comparison with NIST standards

## Evaluation Metrics
- Quantum error rate (QER)
- Classical simulation time (CST)
- Resource efficiency (RE)

## Reproducibility
- Code: https://github.com/smith/quantum-safe
- Data: Publicly available via Zenodo
- Compute: 100 GPU-hours on A100s

**Generated by Cardex Skill: methodology**
```

## Vector Store (Phase 1+)

For semantic search, Cardex uses ChromaDB to store embeddings.

**Location**: `~/.cardex/vector_store/`

**Structure**:
```
~/.cardex/vector_store/
├── chroma.sqlite3           # ChromaDB metadata
└── embeddings/              # Vector data
    └── collection_xyz/
```

**Indexed Content**:
- Paper abstracts
- Summary cards
- User notes

**Not Indexed** (Phase 1):
- Full PDF text (deferred to Phase 2)

## Database Location

**Default**: `~/.cardex/cardex.db`

**Backup Strategy**:
```bash
# SQLite is a single file, easy to backup
cp ~/.cardex/cardex.db ~/Backups/cardex-$(date +%Y%m%d).db

# Or use Git for Markdown files
cd ~/Documents/cardex-library/markdown
git init
git add .
git commit -m "Backup knowledge base"
```

## Data Migration

### From Zotero (Optional Integration)

If Zotero integration is enabled:

```yaml
# config.yaml
zotero:
  enabled: true
  library_path: ~/Zotero/storage
  sqlite_path: ~/Zotero/zotero.sqlite
```

Cardex reads:
- Zotero's `items` table → `papers` table
- Zotero's `itemAttachments` → PDF file paths
- Zotero's `itemNotes` → Initial content for `notes` table

**Note**: This is read-only. Cardex does not modify Zotero's database.

## Performance Considerations

### Database Size Estimates

| Papers | SQLite Size | Markdown Size | Vector Store |
|--------|-------------|---------------|--------------|
| 100    | ~500 KB     | ~5 MB         | ~50 MB       |
| 1,000  | ~5 MB       | ~50 MB        | ~500 MB      |
| 10,000 | ~50 MB      | ~500 MB       | ~5 GB        |

### Query Performance

- **SQLite**: Indexed queries on 10,000 papers < 10ms
- **Full-text search**: Whoosh on 10,000 Markdown files < 100ms
- **Semantic search**: ChromaDB on 10,000 embeddings < 500ms
